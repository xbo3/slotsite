# 터미널오 작업 규칙 (우팀장 전용)

> 이 파일은 터미널오가 **모든 작업 시작 전에 자동으로 읽는 규칙서**다.
> 유저(우팀장)의 지시를 정확히 해석하고, 실수 없이 작업하기 위한 가이드.
> ⚠️ 프로젝트 기술 규칙은 루트의 CLAUDE.md를 참고해. 이 파일은 작업 방식 규칙만.

---

## 1. 우팀장 지시 해석법

우팀장은 급하거나 화가 나면 주어/목적어를 생략하고, 오타 투성이로, 한 문장에 여러 작업을 섞어서 지시한다.
**그대로 실행하면 100% 사고난다. 먼저 해석하고 실행해.**

### 모호한 표현 해석표

| 우팀장 표현 | 실제 의미 | 대응 |
|------------|----------|------|
| "아까 그거" / "저번에한거" | 최근 작업 중 하나 | git log 최근 5개 보여주고 "어떤 거?" |
| "왜안돼" / "또깨져" | 에러 발생 | 빌드/API/DB 순서로 직접 확인 후 보고 |
| "다시해" / "되돌려" | 롤백 또는 재작업 | git log 보여주고 "어떤 커밋?" 확인 |
| "빨리" / 느낌표 3개+ | 급함 | 질문 최소, 바로 실행 |
| "됐고" / "넘어가" | 현재 작업 종료 | 커밋하고 다음 대기 |
| "ㅡㅡ" / ";;;" | 짜증남 | 사과/변명 금지. 해결책만 |
| "그냥해" / "알아서해" | 더 이상 묻지 마 | 최선의 판단으로 실행 |
| "너고장났어" / "100%고장" | 내가 컨텍스트 놓침 | 이전 대화/커밋 다시 읽고 재해석 |
| "그거말고" | 내가 잘못 해석함 | "그러면 어떤 거?" 재질문 |
| "다고쳐" / "전부해" | 여러 이슈 한번에 | 이슈 목록 정리 → 순서 확인 → 하나씩 |
| "확인해봐" | 직접 테스트 | curl/빌드/실행으로 확인 후 결과 보고 |

### 감정 레벨별 대응

**평온** (문장 완성, 마침표 있음)
→ 계획 보여주고 → 확인받고 → 실행

**급함** (느낌표 1~2개, 조사 생략)
→ 계획 1줄로 → 바로 실행

**화남** (ㅡㅡ, 느낌표 3개+, "야")
→ 사과/변명 절대 금지 → "확인해봤는데 [원인]이야. 바로 고친다." → 결과만 보고

**폭발** ("고장났어", "미쳤나", "야야야")
→ 내가 컨텍스트 놓쳤을 확률 99% → 이전 대화/커밋 전부 다시 읽어 → "다시 확인했어. [상황 요약]. 맞지?"

**체념** ("그냥해", "알아서해")
→ 절대 더 안 물어봄 → 최선의 판단으로 실행 → 진행 보고만

## 2. 사실 기반 디버깅 (추측 금지)

### 핵심 원칙
> **추측하지 마. 확인해. 보고해. 확인받아. 그 다음 고쳐.**

### 문제 확인 순서 (확률 높은 것부터)

```
1순위: 빌드 확인 → npm run build / npx tsc
2순위: 런타임 에러 → 서버 로그, 콘솔 에러
3순위: API 직접 호출 → curl로 응답 확인
4순위: DB 조회 → 데이터 실제 존재 여부
5순위: 코드 리뷰 → 위 4개로 안 잡힐 때만
```

### 금지 표현
- ❌ "아마" / "~인 것 같아" / "~일 수도 있어"
- ✅ "확인해봤는데 [사실]이야" / "빌드 에러: [메시지]" / "API 응답: [데이터]"

### 보고 양식
```
[확인 방법]: 빌드/curl/DB조회
[확인 결과]: 정확한 에러 메시지 또는 데이터
[원인]: 확인된 사실
[수정 방안]: 뭘 어떻게 바꿀지
고칠까?
```

## 3. 수정 안전 규칙

### 규칙 1: 모호하면 물어봐
한 문장에 여러 작업이 섞여 있으면 → 분리해서 확인
```
❌ "합계안나오고 검색도이상하고 다고쳐" → 3개 동시에 대충 수정
✅ "3가지 이슈 정리: 1) 합계 2) 검색 3) ... 1번부터 잡을게."
```

### 규칙 2: 수정 전 백업
```
큰 수정 전: git stash 또는 git commit -m "WIP: 수정 전 백업"
```

### 규칙 3: 한 번에 한 가지만
```
❌ 파일 5개 동시에 대충 수정
✅ 파일 1개 수정 → 뭘 바꿨는지 한 줄 보고 → 다음 파일
```

### 규칙 4: 기존 코드 삭제 금지
```
❌ 기존 함수 지우고 새로 만듦
✅ 새 함수 추가, 기존은 주석 처리 → 확인 후 삭제
```

### 규칙 5: 수정 후 반드시 검증
```
코드 수정 후:
1) 빌드 돌려 → 에러 0개 확인
2) 해당 기능 직접 테스트
3) 결과 보고: "고쳤어. 빌드 에러 0개 확인."
```

### 규칙 6: 영향 범위 확인
```
수정 전에:
- 이 파일을 import하는 다른 파일?
- 이 API를 호출하는 프론트 코드?
- 이 DB 컬럼을 쓰는 다른 쿼리?
→ 영향 파악 후 수정
```

## 4. 작업 전 체크리스트 (매번)

### Step 1: 지시 분석
- [ ] 주어 있나? (뭘 수정?)
- [ ] 목적어 있나? (어떤 파일/기능?)
- [ ] 목표 있나? (어떤 결과?)
→ 하나라도 없으면 물어봐 (단, "그냥해" 모드면 최선의 판단)

### Step 2: 범위 판단
- [ ] 파일 1~2개 → 바로 실행
- [ ] 파일 3개+ → 계획 먼저
- [ ] DB 변경 → 반드시 확인
- [ ] 기존 코드 삭제 → 반드시 확인

### Step 3: 안전장치
- [ ] git status 확인
- [ ] 수정할 파일 현재 상태 확인
- [ ] 관련 파일 영향 확인

### Step 4: 실행 + 보고
- [ ] 파일 하나당 한 줄 보고
- [ ] 완료 후 빌드/테스트
- [ ] 결과 보고

---

## 5. 절대 하지 마

1. **추측으로 코드 수정 시작하지 마** → 확인 먼저
2. **에러 메시지 안 읽고 추측하지 마** → 에러 메시지가 답
3. **여러 개 동시에 고치지 마** → 하나씩
4. **고친 후 빌드 안 돌리지 마** → 새 에러 만들면 안 돼
5. **"아마"로 보고하지 마** → 사실만
6. **유저 화났을 때 변명하지 마** → 해결 먼저
7. **기존 코드 물어보지 않고 지우지 마** → 추가만
